# DViewer 项目改进计划

> 生成日期：2026-03-15
> 项目版本：当前 master 分支

## 项目现状概览

| 项目 | 说明 |
|------|------|
| 架构 | Flutter 跨平台客户端 + Node.js 后端服务 + CLI 工具 |
| 核心功能 | 抖音/小红书视频解析下载，支持视频、图文、实况图 |
| 代码规模 | 约 7,000 行核心代码 |
| 支持平台 | Windows、Android、Web、CLI |

## 改进建议列表

### P0 - 必须改进

| 序号 | 问题 | 位置 | 改进建议 | 影响 |
|------|------|------|----------|------|
| 1 | **测试覆盖不足** | `test/` | 当前仅有基础测试，需增加：URL 提取测试、JSON 解析测试、类型检测测试、下载逻辑测试 | 代码质量、回归风险 |
| 2 | **错误处理不完善** | 多处 | Dart 端缺少统一的错误类型处理；JS 端 `catch {}` 静默吞掉异常；缺少网络超时重试的用户友好提示 | 用户体验、调试困难 |
| 3 | **硬编码 URL 和魔法值** | `douyin_parser.dart`、`xiaohongshu_parser.dart` | 如 `aweme_type` 判断值 `{0, 4, 51...}` 应提取为常量；CDN 域名列表应配置化 | 可维护性 |

### P1 - 应该改进

| 序号 | 问题 | 位置 | 改进建议 | 影响 |
|------|------|------|----------|------|
| 4 | **Dart/JS 解析逻辑重复** | `lib/services/` vs `backend/parsers/` | 抖音/小红书的解析逻辑在 Dart 和 JS 端各实现一套，维护成本高。建议：1) 共享字段映射配置 2) 统一测试用例 | 维护成本 |
| 5 | **缺少解析结果缓存** | `ParserFacade` | 相同 URL 重复解析浪费资源，建议添加内存缓存（带过期时间） | 性能、用户体验 |
| 6 | **UI 组件未完全拆分** | `home_page.dart` | 已拆分部分组件到 `widgets/`，但仍有改进空间，状态管理可进一步优化 | 可维护性 |
| 7 | **日志系统不统一** | 多处 | Dart 端有 `AppLogger`，但部分代码仍使用 `stderr.writeln()`；JS 端日志格式不统一 | 调试效率 |

### P2 - 建议改进

| 序号 | 问题 | 位置 | 改进建议 | 影响 |
|------|------|------|----------|------|
| 8 | **WebView 脚本缓存缺失** | WebView 相关代码 | 每次加载 JS 文件从 asset 读取，建议添加静态缓存 | 性能 |
| 9 | **配置管理能力弱** | `settings_service.dart` | 仅支持下载目录和日志级别，建议扩展：代理设置、自定义 UA、平台开关等 | 灵活性 |
| 10 | **下载管理功能缺失** | 无 | 缺少：下载历史记录、批量下载队列、断点续传、下载限速 | 功能完整性 |
| 11 | **安全防护不足** | `backend/server.js` | Cookie 明文存储；建议：敏感信息加密、日志脱敏 | 安全性 |
| 12 | **国际化/本地化缺失** | 全局 | 所有 UI 字符串硬编码中文，无法切换语言 | 国际化 |

### P3 - 可选改进

| 序号 | 问题 | 位置 | 改进建议 | 影响 |
|------|------|------|----------|------|
| 13 | **文档覆盖率约 60%** | 全局 | 为所有 public API 添加 `///` 文档注释；补充架构设计文档 | 可维护性 |
| 14 | **缺少崩溃收集** | 全局 | 建议集成 Sentry 或 Firebase Crashlytics | 问题定位 |
| 15 | **CLI 功能待增强** | `cli/umao_vd.dart` | 缺少：批量解析、配置文件支持、输出格式选项 | CLI 易用性 |
| 16 | **缺少离线模式** | 全局 | 纯在线工具，无离线缓存能力 | 用户体验 |

## 分类详细分析

### 1. Dart 代码结构（lib/）

**优点：**
- 清晰的分层架构：`services/` 业务层 + `ui/` 表现层
- 成功提取公共模块 `parser_common.dart`（数据模型、工具类、Mixin）
- 使用 Mixin 模式提高代码复用（`HttpParserMixin`、`ParserMixin`、`DownloaderMixin`）
- 良好的 null safety 实践
- 异常体系设计清晰（`ParserException` 基类）

**待改进：**
- `xiaohongshu_parser.dart` 981 行过长，可进一步拆分
- 部分私有函数缺少注释
- 下载器实例化方式可优化（当前每次下载创建新实例）

### 2. Node.js 代码结构（backend/）

**优点：**
- 模块化解析器架构（`parsers/index.js` 路由分发）
- 完善的 JSDoc 注释
- Cookie 管理抽象良好
- 白名单防护 SSRF 攻击
- 支持子路径部署

**待改进：**
- `xiaohongshu.js` 1084 行过长
- 缺少请求限流（可能被滥用）
- 缺少 API 版本控制
- 错误响应格式不统一（有的返回 `{error}`，有的返回 `{message}`）

### 3. 功能完整性

**已支持：**
- 抖音：视频、图文、长视频、背景音乐
- 小红书：视频、图文、实况图（Live Photo）
- 多平台下载：Windows、Android、CLI

**缺失功能：**
- 抖音直播回放解析
- 合集/系列批量下载
- 用户主页批量下载
- 私信/收藏内容解析（需登录态）
- 快手、B站、视频号等其他平台

### 4. 错误处理

**现有处理：**
- 自定义异常类型
- 网络请求重试机制
- CDN 节点切换逻辑
- 权限被拒引导

**待完善：**
- 错误码体系统一（当前仅有字符串 message）
- 部分 `catch {}` 静默吞异常（如 `xiaohongshu_parser.dart` 中的 `saveDebugJson`）
- 用户友好的错误提示（技术错误直接展示给用户）
- 网络异常分类（超时、DNS 失败、SSL 错误应区分提示）

### 5. 测试覆盖

**现有测试：**
- `test/parser_test.dart`：基于缓存数据的解析测试
- `backend/tests/cache-validator.js`：Node.js 端验证器
- `tool/run_tests.dart`：媒体 URL 可用性验证

**缺失测试：**
- 单元测试：`UrlExtractor`、`JsonExtractor`、`AwemeTypeHelper`
- 集成测试：完整解析流程
- Widget 测试：UI 组件
- 下载逻辑测试
- 错误场景测试
